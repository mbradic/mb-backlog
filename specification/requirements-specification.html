1. Executive Summary

mb-backlog is a personal, single-user web application for maintaining a backlog of tasks. It provides a simple interface to add, edit, and remove backlog items so that the user can keep track of work that still needs to be done. The application is intentionally lightweight: a plain HTML/CSS/JavaScript frontend talks to a Node.js backend with no external frameworks or libraries, backed by a local SQLite database.

Key Objectives:

  • Provide a simple way to capture backlog items (tasks) as they come up
  • Allow items to be edited and removed as priorities and plans change
  • Keep the implementation dependency-free and easy to run and maintain

2. Project Overview

2.1 Background

The user needs a straightforward place to keep a running backlog of personal tasks, instead of relying on scattered notes or third-party tools. mb-backlog addresses this by offering a small, self-hosted web application dedicated solely to backlog management.

2.2 Scope

In Scope:

  • Viewing the list of backlog items
  • Adding new backlog items
  • Editing existing backlog items
  • Removing (deleting) backlog items

Out of Scope:

  • Multi-user support, accounts, or authentication
  • Sharing or collaboration features
  • Mobile native apps
  • Integrations with third-party task tools

2.3 Target Users

  • Owner: The single user of the application, who both develops and uses it to manage their personal backlog.

4. Non-Functional Requirements

4.1 Performance

  • Response time for list, view, add, edit, and delete operations: under 200ms on localhost
  • Initial page load: under 1 second when served locally

4.2 Scalability

The application is designed for a single user with a modest number of backlog items (expected up to a few thousand rows). SQLite is sufficient for this scale; no horizontal scaling or clustering is required.

4.3 Security

  • Authentication: Not required for v1.0 — the application is intended to run locally for a single user
  • Network exposure: The backend should bind to localhost by default and not be exposed to the public internet without additional hardening
  • Input handling: All user input must be validated on the server and persisted using parameterized SQL queries to prevent SQL injection
  • Output handling: Any user-supplied text rendered in the UI must be escaped to prevent XSS

4.4 Reliability & Availability

  • Uptime SLA: None — personal, locally-run application
  • Data persistence: Backlog data is stored in a local SQLite file; the user is responsible for periodically backing up this file

4.5 Usability

  • Browser compatibility: Latest versions of Chrome, Firefox, and Edge
  • The UI should be simple and usable on a single desktop screen size; mobile/responsive layout is not required for v1.0

4.6 Maintainability

  • Technology stack: Plain JavaScript, HTML, and CSS on the frontend; Node.js built-in modules only on the backend — no external libraries or frameworks
  • Testing: Verified via the manual test scenarios documented in Section 3; no automated test framework is required for v1.0

5. Technical Specifications

5.1 Architecture

mb-backlog follows a simple client-server architecture. A static frontend (HTML, CSS, and vanilla JavaScript) runs in the browser and communicates with a Node.js backend over a small JSON HTTP API, built using Node's built-in http module (no framework such as Express). The backend reads from and writes to a local SQLite database file.

5.2 Technology Stack

Layer Technology Version Justification
Frontend Vanilla HTML / CSS / JavaScript N/A Project constraint: no external frontend libraries or frameworks
Backend Node.js (built-in modules only) Latest LTS Project constraint: no external backend frameworks (e.g. Express)
Database SQLite 3.x Lightweight, file-based, and sufficient for a single-user application with no external DB server

5.3 Database Schema (Summary)

A single table stores all backlog items:

Column Type Notes
id INTEGER PRIMARY KEY AUTOINCREMENT Unique item identifier
title TEXT NOT NULL Required short title of the item
description TEXT Optional free-text details

5.4 API Specifications

Method Endpoint Description
GET /api/items Returns the list of all backlog items (FR-001)
GET /api/items/:id Returns the details of a specific backlog item (FR-005)
POST /api/items Creates a new backlog item (FR-002)
PUT /api/items/:id Updates an existing backlog item (FR-003)
DELETE /api/items/:id Deletes a backlog item (FR-004)

5.5 Integration Points

None. mb-backlog is a fully self-contained, single-user, locally-run application with no third-party service integrations in v1.0.

6. Constraints & Assumptions

6.1 Constraints

  • Budget: None — personal side project
  • Timeline: No fixed deadline
  • Resource: Single developer (the user), working in spare time
  • Technical: No external frontend or backend libraries/frameworks are allowed; Node.js built-in modules and SQLite must be used
  • Regulatory: None

6.2 Assumptions

  • The application runs on a single machine and is accessed by a single user, so no authentication is needed for v1.0
  • Node.js is installed on the machine that runs the backend
  • The volume of backlog items will remain small enough for SQLite to handle comfortably without tuning

7. Dependencies & Risks

7.1 External Dependencies

  • Node.js runtime: Required to run the backend; without it the application cannot start
  • SQLite: Required for data persistence; without it backlog data cannot be stored

7.2 Known Risks

Risk Probability Impact Mitigation Strategy
Implementing HTTP routing and request handling without a framework increases development effort and risk of bugs Medium Medium Keep routing logic minimal and consistent; manually verify each endpoint against the test scenarios in Section 3
No third-party validation library is used, so input validation is fully manual Medium Medium Validate and sanitize all input server-side; use parameterized SQL queries for every database call
Local SQLite file could be lost or corrupted with no automated backup Low Medium Document a manual backup process for the SQLite file
