1. Executive Summary

mb-backlog is a personal web application for managing a single user's task backlog. It allows the user to capture, track, and complete tasks through a simple browser-based interface backed by a local Node.js server and a SQLite database. The application is intentionally minimal: it is built with vanilla JavaScript and Node.js core modules only, with no external frameworks or npm dependencies, and is intended to run on the owner's own machine.

Key Objectives:

  • Provide a fast, no-friction way to capture and manage personal tasks in a single backlog
  • Allow the backlog to be manually reordered so the most important work is always visible at the top
  • Track the status of each task (Backlog, In Progress, Done) throughout its lifecycle
  • Keep the technology footprint minimal (pure JS/Node.js, SQLite, no external libraries)

2. Project Overview

2.1 Background

The owner currently lacks a single, lightweight place to keep track of personal tasks and their relative order of importance. Existing note-taking tools and to-do apps either require external accounts, pull in heavy frameworks/dependencies, or don't fit the exact workflow of maintaining an ordered backlog. mb-backlog addresses this by providing a small, self-hosted, dependency-free application focused purely on managing a personal task backlog.

2.2 Scope

In Scope:

  • Web-based UI for creating, viewing, editing, and deleting backlog items
  • Status tracking for each item (Backlog, In Progress, Done)
  • Manual reordering of items within the backlog
  • Filtering the backlog list by status
  • Persistent local storage of all data in a SQLite database file
  • A minimal JSON API served by a Node.js HTTP server

Out of Scope:

  • Multi-user support, accounts, or authentication
  • Cloud hosting, remote access, or multi-device synchronization
  • Native mobile applications
  • Reminders, notifications, or calendar integration
  • Third-party integrations of any kind

2.3 Target Users

  • Owner (single user): The sole user of the application, who creates, organizes, and completes their own personal tasks.

4. Non-Functional Requirements

4.1 Performance

  • Backlog list load time: under 200 milliseconds for up to 1,000 items on the local machine
  • Add/edit/delete/reorder operations: perceived as instantaneous (under 100 milliseconds) since all processing is local
  • Single concurrent request at a time is sufficient; no concurrency tuning required

4.2 Scalability

As a single-user, locally run application, mb-backlog does not need to scale to multiple users or servers. It should comfortably handle a personal backlog of up to a few thousand items without noticeable performance degradation, using simple SQLite indexing on status.

4.3 Security

  • Authentication: none required; the application is intended for local, single-user use only and must not be exposed to the public internet
  • Authorization: not applicable (single user, no roles)
  • Data Handling: all SQL queries must use parameterized statements to prevent SQL injection; all user input rendered in the UI must be safely escaped to prevent cross-site scripting (XSS)
  • Data Encryption: not required for v1, since data resides only in a local SQLite file on the owner's machine; the server should bind to localhost only
  • Compliance: none applicable (personal, non-commercial, single-user data)

4.4 Reliability & Availability

  • Uptime SLA: not applicable; the server runs on demand when the owner starts it
  • Mean Time To Recovery (MTTR): restarting the Node.js process should restore full functionality within seconds
  • Backup Strategy: the owner is responsible for periodically copying the SQLite database file to back it up; no automated backup is required for v1

4.5 Usability

  • Browser compatibility: latest two versions of Chrome, Edge, and Firefox
  • Accessibility: basic keyboard navigability for forms and buttons; no formal WCAG certification required for v1
  • Response to user interactions: under 100 milliseconds for UI feedback (e.g., button clicks, status changes)

4.6 Maintainability

  • Code documentation: key modules and API endpoints should have brief comments describing their purpose
  • Testing: manual test scenarios as documented in Section 3; no formal automated coverage target for v1
  • Technology stack: intentionally minimal (vanilla JavaScript, Node.js core modules, SQLite) to keep the codebase easy to understand and maintain without dependency upgrades

6. Technical Specifications

6.1 Architecture

mb-backlog uses a simple client-server architecture running entirely on the owner's local machine. A Node.js process, built using only Node's built-in http module (no web frameworks), serves static frontend assets (HTML, CSS, vanilla JavaScript) and exposes a small JSON API under /api. The API reads and writes to a single SQLite database file using Node's built-in node:sqlite module, with no external npm packages. The browser-based frontend uses vanilla JavaScript to call the API and render the backlog list, with no frontend frameworks or build tooling.

6.2 Technology Stack

Layer Technology Version Justification
Frontend Vanilla JavaScript, HTML5, CSS3 N/A (no framework) Meets the requirement for a pure JS solution with no external libraries and minimal footprint
Backend Node.js (built-in http module only) Node.js 22 LTS or later Pure Node.js with no external frameworks, as required; Node 22+ provides the built-in node:sqlite module
Database SQLite (single-file database) SQLite 3, accessed via node:sqlite Lightweight, file-based, zero-configuration storage suitable for a single-user local app

6.3 Database Schema (Summary)

A single table, backlog_items, stores all data: id (primary key), title, description, status (Backlog / In Progress / Done), sort_order (integer, for manual reordering), created_at, and updated_at. No other tables are required for v1, since the application is single-user and has no relational entities beyond backlog items.

6.4 API Specifications

The backend exposes the following minimal JSON endpoints under /api:

  • GET /api/items — list all backlog items; supports an optional status query parameter for filtering, ordered by sort_order
  • POST /api/items — create a new backlog item (title required; description optional)
  • GET /api/items/:id — retrieve a single backlog item
  • PUT /api/items/:id — update an existing item's title, description, or status
  • DELETE /api/items/:id — permanently delete an item
  • PUT /api/items/reorder — update the sort_order of one or more items after a drag/reorder action

6.5 Integration Points

  • None. mb-backlog is a fully standalone application with no external system integrations in v1.

7. Constraints & Assumptions

7.1 Constraints

  • Budget: None; this is a personal, non-commercial project
  • Timeline: No fixed deadline
  • Resource: Single developer (the owner), working part-time
  • Technical: Must be implemented using pure JavaScript and Node.js core modules only, with no external frameworks or npm libraries; must use SQLite for persistence
  • Regulatory: None applicable; all data is personal and stored locally

7.2 Assumptions

  • The application runs locally on the owner's own machine and is not exposed to the public internet
  • The owner has Node.js 22 LTS or later installed, providing built-in node:sqlite support
  • Only a single browser session accesses the application at any given time
  • Multi-device synchronization is not required for v1

8. Dependencies & Risks

8.1 External Dependencies

  • Node.js runtime (version 22 LTS or later): required to run the server and access the built-in node:sqlite module
  • SQLite (bundled via node:sqlite): required for all data persistence

8.2 Known Risks

Risk Probability Impact Mitigation Strategy
The built-in node:sqlite module's API changes between Node.js versions since it is relatively new Medium Medium Pin the project to a specific Node.js LTS version and document it clearly; re-test after any Node.js upgrade
Data loss due to absence of an automated backup mechanism Low High Document a manual backup process (copying the SQLite file) and recommend the owner do so periodically
